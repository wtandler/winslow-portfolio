## The Problem

The failures that hurt a portfolio usually come from questions nobody thought to ask. Two teams running on assumptions that quietly contradict each other. A dependency getting bigger while everyone treats it as settled. A risk that spans three orgs, so no single person sees its whole shape. The information exists. Nobody connected it in time.

Status tools don't help, because they wait. They sit there until someone asks the right question, and the right question is the one you don't know to ask. By the time it surfaces in a meeting with five VPs and twenty managers, it's too late. The signal that would have caught it is scattered: in a few long-tenured heads, in slide decks that took hours to build, in back-to-back calls nobody can cancel. Different teams describe the same work in different words and different tools, so you get status without clarity.

## The Insight

The fix is to stop waiting for the user to ask.

Threadbase runs like a team of analysts that never stops reading. Agents work across the portfolio continuously, comparing what one team said against what another assumed, tracking how dependencies move, watching for assumptions that expired without anyone saying so. Instead of answering questions, it asks them first and puts what it found in front of you to confirm or act on.

For that to be credible inside an organization, every claim has to survive scrutiny. The core design rule is evidence-first: nothing enters the model of the world without a link back to the exact document, page, and region it came from, and nothing the AI asserts counts as truth until a person accepts it or it clears a configurable confidence bar.

## The Approach

I started with a chat interface, because "AI product" basically implies chat. That was a mistake. If someone has to ask the system what's going on, you've added back the exact friction it was meant to remove.

So Threadbase is built around a persistent model of the portfolio (initiatives, owners, dependencies, risks, decisions, and the facts behind them) with agents working against it in the background. When something changes, the knock-on effects show up early. When two things conflict, it flags the conflict side by side, with both sources. That choice forced the real engineering: a document pipeline that survives corporate slides, retrieval across a knowledge graph, an agent layer with durable memory, and the evals and governance that make any of it trustworthy enough to act on.

## How It's Built

A six-stage evidence pipeline (ingest, embed, extract, verify, analyze, publish) runs on Azure Container Apps against PostgreSQL. No ORM: raw parameterized SQL over 60+ versioned migrations, applied on container boot and serialized across replicas with advisory locks. Around 2,000 tests and eight eval suites gate changes.

### Document understanding

Corporate slides are brutal: dense layouts, nested tables, icons that matter, "this one dot means red." After burning through the open-source OCR stacks, I landed on Azure Content Understanding plus a multimodal vision pass that emits structured facts with bounding boxes on a normalized grid. Every extracted claim renders as a highlight on the source page, so a reviewer never has to take the AI's word for anything.

### The fact ledger

Extraction writes subject-predicate-object claims with confidence scores, qualifiers, temporal windows, and full lineage to their source chunk. The verify stage detects conflicts with already-accepted claims and routes them to the right owner by walking the entity graph. Reviewers accept, edit (with a written reason), reject, escalate, or keep both sides with explicit effective-date windows, and every mutation is audited. The auto-accept threshold is a runtime setting, so where the line falls between machine and human is a policy an admin can move at any time.

### Graph-augmented retrieval

The knowledge graph is the core of the product, but it lives in Postgres with pgvector rather than a graph database; a relationship-aware relational model with vector indexes gets most of the benefit with far less operational drag. On top of it: cross-document entity resolution that merges the same initiative when three teams describe it three ways, community detection for portfolio-wide questions, multi-hop traversal so the agent can walk relationships instead of hoping similarity search finds the path, triplet embeddings that index the facts themselves, and a query router that decomposes hard questions and picks the retrieval mode per sub-question. In chat, the agent plans those sub-questions, calls the tools, and answers with rendered components (fact tables, timelines, conflict pairs) and citations harvested from the actual results rather than the model's memory.

### The agent layer

The newest piece treats synthesis as an agent reasoning over the corpus instead of a pile of scheduled detectors. It has durable, versioned memory (beliefs it can recall, write, and supersede), and every conclusion carries a reasoning trace. After each upload, a reflective loop re-reads what changed, proposes belief updates, and updates the recommendation queue that drives the daily briefing, behind a feature flag and with idempotency fingerprints so retries can't double-write. It shipped cautiously: shadow mode first, evals gating promotion, kill switch on the way out.

### Determinism first

The rule I'd defend hardest in a review: LLM output is never load-bearing where a deterministic path can do the job. The prose explaining each pending fact to reviewers is built by a template from structured anchors, so it can't fabricate and reads identically every run; an optional LLM polish pass sits behind a flag with a validator, a circuit breaker, and a fallback to the template. Guardrails live in code: prompt-injection fencing on any user text that reaches a model, a disabled commit button on low-confidence calls, and every model behind an OpenAI-compatible provider layer so swapping one is a config change. Cheap models triage; the expensive calls are reserved for synthesis.

### Evals and production

Eight eval suites cover extraction F1, retrieval quality, portfolio-level Q&A graded on answer quality, agent behavior, predicate classification, and reviewer prose, the last with an LLM-as-judge rubric over deterministic validators. Prompt and model changes land only when the suites say they didn't regress; the same harness ran an embedding-provider tournament and a reranking study before either touched production. Infrastructure is Bicep: Container Apps behind a VNet, managed identity instead of connection strings, Postgres with public access disabled, operational scripts running as an allowlisted job inside the VNet. It's a side project, but it's operated like production software.

## Building with AI

Threadbase is the largest thing I've built, and most of it is written with AI. By the time I started it, generating code fast was no longer the problem. AI makes you faster in whatever direction you were already pointed, and if the direction is vague you get vague output, just sooner. Threadbase is where I stopped improvising prompts and built a repeatable workflow around the model instead.

Instead of asking a model to "go fix this," I route the work through fixed procedures: one for scoping a change, one for exploring the codebase before touching it, one for reviewing a diff, one for debugging. A few things hold it together:

* **A knowledge graph of the code.** Before any change, the workflow checks the blast radius and the cross-module connections I wouldn't have thought to look for. Nothing gets called "isolated" until the graph agrees.
* **Planning before code.** Major features start as a written plan, and there are dozens now. Scoping happens on paper, and adjacent work gets logged instead of quietly absorbed.
* **Tests from the acceptance criteria, not the implementation.** If someone reimplemented a feature differently but correctly, the tests should still pass.
* **Project rules that don't drift.** Naming, SQL, architecture, providers, deployment: written down once, applied every time. The eval suites hold the AI's own output to the same bar.

The point of all of it is consistency. A model will happily build the wrong thing very fast, and the procedures catch that before it compounds. The workflow didn't stay here either: it's how I've built every project since.

## What I Learned

* **Adoption depends on effort.** If using the tool takes work, people fall back to what they already do.
* **Use the data people already create.** Don't ask for "one more update." Extract value from what already exists.
* **A knowledge graph doesn't require a graph database.** The graph is the model that matters. A solid relational model plus vector search carries most of it, minus the operational drag.
* **Provenance is what earns action.** Spatial citations, audit trails, deterministic fallbacks, and human review checkpoints are the reason anyone inside an org acts on what the system says.
* **AI moves the bottleneck.** Once code generation is fast, the limiting factor is knowing exactly what you want built, and proving with evals that you got it.
